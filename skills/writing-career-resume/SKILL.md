# 경력기술서 MD 파일 작성 가이드

## Purpose

사용자의 경력 정보를 받아 정해진 MD 포맷으로 경력기술서를 작성합니다. 심사자가 쉽게 파악할 수 있도록 간결하고 임팩트 있는 문서를 생성합니다.

## When to Use

- 경력기술서, 이력서를 작성할 때
- 채용 지원을 위한 문서 작성 시
- 프로젝트 경력을 정리할 때
- 지원 회사의 자격요건/우대사항에 맞춰 경력서를 수정할 때

---

## 작성 규칙

### 1. 경력 정렬 순서

**최신순 내림차순으로 작성**
- 가장 최근 경력이 맨 위에 위치
- 회사 내 프로젝트도 최신순으로 정렬

### 2. 기간 표기

**형식**: `2023. 03 ~ 2023.10 (7개월)`

- 반드시 기간 옆에 괄호로 총 기간 표시
- 심사자가 한눈에 프로젝트 규모 파악 가능

**기간 설정 가이드라인**:
- 기간 미지정 시: 최대 1년 내 적정 기간 산정
- 너무 짧은 기간: 빠르게 종료된 인상
- 너무 긴 기간: 루즈하게 진행된 인상

### 3. 성과와 역할 구분

| 구분 | 의미 | 작성 방식 |
|------|------|-----------|
| 성과 | 팀 전체의 성과 | "~를 해서 ~결과로 ~을 달성함" |
| 역할 | 개인의 기여 | 구체적인 기술적 구현 내용 |

**성과 작성 공식**:
```
[수행 내용] + [결과/효과] + [달성 목표]
```

예시:
- 챗봇 서비스를 사내 솔루션에 런칭함으로써 CS 30% 감소

### 4. 문체 규칙

**개조식 필수** (서술형 금지)

| 금지 | 권장 |
|------|------|
| ~하였습니다 | ~하였음 |
| ~했습니다 | ~처리함 |
| ~입니다 | ~임 |
| ~개발했습니다 | ~개발 |

**간결성 원칙**:
- 불필요한 부사 제거
- 수식어 최소화
- 핵심 내용만 기술

### 5. 기술 스택 정렬

**부각되는 기술 순 내림차순**
- 주력 언어 > 주력 프레임워크 > 부가 기술
- 해당 프로젝트에서 핵심적으로 사용한 기술 우선

### 6. 프로젝트 선별

**포인트가 되는 프로젝트 위주**
- 30년 경력이라도 모든 프로젝트 나열 금지
- 임팩트 있는 프로젝트 선별
- 지원 직무와 연관성 높은 프로젝트 우선

### 7. 지원 회사 맞춤

**자격요건/우대사항 첨부 시**:
- 해당 요건에 맞는 경험 강조
- 관련 기술 스택 상위 배치
- 연관 프로젝트 우선 기술

## MD 포맷 템플릿

```markdown
# 경력

### (주) 회사명

*위치 – (입사년월 - 현재/퇴사년월)*

| 기간 | YYYY. MM ~ YYYY. MM (N개월/N년) |
| --- | --- |
| 성과 | [팀 성과: ~를 해서 ~결과로 ~달성함] |
| 역할 | - 구체적 기술 구현 내용 1
- 구체적 기술 구현 내용 2
- 구체적 기술 구현 내용 3 |
| 기술 | - Language: 주력언어
- Framework: 주력프레임워크
- Database: DB
- 기타 기술스택 |

---

# 기술

### 주력 기술명

해당 기술에 대한 숙련도와 활용 경험을 2-3문장으로 기술

### 보조 기술명

해당 기술에 대한 경험 기술

# 기술 (not used)

현재 사용하지 않는 기술 (선택)

---

# 문서 / 도구

### 도구명

활용 경험 기술

---

# 언어

### 한국어

원어민

### 외국어

수준 - 자격증

---

# 학력

## 전공

졸업년도

학교명 - 위치

---

# 이수 / 교육

## 교육명

이수년도

교육기관

---

# 자격증

## 자격증명

취득년도

---

# 연락처

이메일
LinkedIn
블로그
GitHub
```

## 작성 체크리스트

- [ ] 경력이 최신순으로 정렬되어 있는가?
- [ ] 모든 기간에 (N개월/N년) 표기가 있는가?
- [ ] 성과가 "~해서 ~결과로 ~달성함" 형식인가?
- [ ] 역할이 개인의 구체적 기여인가?
- [ ] 개조식으로 작성되었는가? (서술형 없음)
- [ ] 불필요한 부사/수식어가 제거되었는가?
- [ ] 기술 스택이 중요도순으로 정렬되었는가?
- [ ] 핵심 프로젝트만 선별되었는가?
- [ ] (해당시) 지원 회사 요건에 맞춰 작성되었는가?

## 예시

### 좋은 예시

```markdown
| 기간 | 2025. 03 ~ 2025.10 (7개월) |
| --- | --- |
| 성과 | 챗봇 서비스를 사내 솔루션에 런칭함으로써 CS 30% 감소 |
| 역할 | - 실시간 응답 처리를 위한 SSE 스트림 처리 구현
- VectorDB Collection 및 Index 구성 처리
- Kubernetes 시스템 구조도 설계 |
| 기술 | - Language: Python
- Framework: FastAPI
- Database: MongoDB, Milvus VectorDB |
```

### 나쁜 예시

```markdown
| 기간 | 2025. 03 ~ 2025.10 |
| --- | --- |
| 성과 | 챗봇 서비스를 개발했습니다 |
| 역할 | - 저는 실시간 응답 처리를 아주 열심히 구현하였습니다
- 매우 효율적인 VectorDB를 성공적으로 구축하였습니다 |
| 기술 | Python, FastAPI 등 |
```

**문제점**:
- 기간에 총 기간 표기 없음
- 성과가 구체적 결과 없음
- 서술형 문체 사용
- 불필요한 부사 (아주, 매우, 성공적으로)
- 기술 스택 구조화 안됨
